```typescript
// Standalone migration runner for CI-02.
// Uses drizzle-orm/node-postgres/migrator (prod dep) — NOT drizzle-kit (dev dep, pruned in runtime image).
// Invoked by the open-seo-migrate compose service during deploy, BEFORE new app containers go live.

import pg from "pg";
import { drizzle } from "drizzle-orm/node-postgres";
import { migrate } from "drizzle-orm/node-postgres/migrator";

async function main(): Promise<void> {
  const connectionString = process.env.DATABASE_URL;
  if (!connectionString) {
    console.error("[migrate] DATABASE_URL is required");
    process.exit(1);
  }

  const pool = new pg.Pool({ connectionString });
  const db = drizzle(pool);

  console.log("[migrate] applying migrations from ./drizzle ...");
  try {
    await migrate(db, { migrationsFolder: "./drizzle" });
    console.log("[migrate] migrations applied successfully");
  } catch (err) {
    console.error("[migrate] migration failed:", err);
    await pool.end();
    process.exit(1);
  }

  await pool.end();
  process.exit(0);
}

main();
```

Notes on WHY:
- `drizzle-kit migrate` is NOT used because `drizzle-kit` is in `devDependencies` and `pnpm prune --prod` in Dockerfile.vps removes it from the runtime image.
- `drizzle-orm/node-postgres/migrator` IS in the runtime image (prod dep) and is the documented programmatic migration API.
- `migrationsFolder: "./drizzle"` matches the WORKDIR-relative path where `COPY --from=builder /app/drizzle ./drizzle` lands in the runtime image.
- We explicitly `await pool.end()` before `process.exit(0)` so the container exits cleanly with `restart: "no"`.

```yaml
  # ===== MIGRATIONS (one-shot, behind `migrate` profile — CI-02) =====
  # Invoked by CI as:
  #   docker compose -f docker-compose.vps.yml --profile migrate run --rm open-seo-migrate
  # This service is NOT started by a plain `docker compose up` because of the `profiles` key.

  open-seo-migrate:
    image: teveroseo/open-seo:latest
    # Reuses the image built by the `open-seo` service. If CI runs `up -d --build open-seo`
    # before `run --rm open-seo-migrate`, the fresh image is already tagged and ready.
    profiles: ["migrate"]
    restart: "no"
    command: ["node", ".output/migrate-entry.mjs"]
    environment:
      NODE_ENV: production
      DATABASE_URL: "postgresql://open_seo_user:${OPEN_SEO_DB_PASSWORD}@postgres:5432/open_seo"
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - teveroseo-net
```

Do NOT modify any other service, volume, network, or top-level field.

Why these choices:
- `profiles: ["migrate"]` → the service is inert for `docker compose up` / healthchecks / dependency ordering, but opt-in via `--profile migrate run --rm`.
- `restart: "no"` → CI expects the container to exit after migrations; restart policies would re-run it forever.
- `image: teveroseo/open-seo:latest` → reuses the tag produced by the `open-seo` build; no duplicate build context.
- DATABASE_URL matches the exact string used by `open-seo` service (line 61) so migrations hit the same database with the same credentials.
- `depends_on postgres service_healthy` → respects postgres startup, satisfies CI-02 ordering constraint (migrate only when DB is up).
