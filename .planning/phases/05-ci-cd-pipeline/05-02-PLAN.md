```yaml
# deploy-vps.yml — open-seo-main auto-deploy to VPS (CI-01, CI-02, CI-03, CI-04, CI-05)
#
# REQUIRED GitHub Actions repository secrets (configure under Settings → Secrets → Actions):
#   VPS_HOST              — IP or hostname of the VPS
#   VPS_USER              — dedicated Linux user "deploy" (docker group member, CI-04)
#   VPS_SSH_PRIVATE_KEY   — ed25519 PRIVATE key (public half lives in /home/deploy/.ssh/authorized_keys on the VPS)
#   KNOWN_HOSTS           — output of `ssh-keyscan -H <VPS_HOST>` — pins the host key (CI-03)
#
# VPS setup prerequisites (run once, by hand, as root):
#   useradd -m -s /bin/bash deploy
#   usermod -aG docker deploy
#   mkdir -p /home/deploy/.ssh && chmod 700 /home/deploy/.ssh
#   echo '<PUBLIC_KEY>' >> /home/deploy/.ssh/authorized_keys
#   chmod 600 /home/deploy/.ssh/authorized_keys
#   chown -R deploy:deploy /home/deploy/.ssh
#   sudo -u deploy git clone git@github.com:<org>/TeveroSEO.git /home/deploy/TeveroSEO
#
# Deploy flow on the VPS:
#   1. git pull origin main
#   2. docker compose --profile migrate run --rm open-seo-migrate   (CI-02, runs BEFORE container swap)
#   3. docker compose up -d --build open-seo open-seo-worker        (only these two services rebuild)

name: deploy-vps

on:
  push:
    branches: [main]
    paths:
      - "open-seo-main/**"
      - "docker-compose.vps.yml"
      - ".github/workflows/deploy-vps.yml"
  workflow_dispatch: {}

concurrency:
  group: deploy-vps-open-seo
  cancel-in-progress: false

env:
  VPS_REPO_PATH: /home/deploy/TeveroSEO
  COMPOSE_FILE: docker-compose.vps.yml

jobs:
  deploy:
    name: Deploy open-seo to VPS
    runs-on: ubuntu-latest
    timeout-minutes: 20

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Configure SSH client with pinned host key (CI-03, CI-05)
        shell: bash
        env:
          SSH_PRIVATE_KEY: ${{ secrets.VPS_SSH_PRIVATE_KEY }}
          KNOWN_HOSTS: ${{ secrets.KNOWN_HOSTS }}
        run: |
          set -euo pipefail
          mkdir -p ~/.ssh
          chmod 700 ~/.ssh

          # Write pinned host key BEFORE any SSH call — this is the CI-03 guarantee.
          # KNOWN_HOSTS secret must be the output of `ssh-keyscan -H <VPS_HOST>`.
          printf '%s\n' "$KNOWN_HOSTS" > ~/.ssh/known_hosts
          chmod 600 ~/.ssh/known_hosts

          # Load the private key into an ssh-agent (not written to disk).
          eval "$(ssh-agent -s)"
          printf '%s\n' "$SSH_PRIVATE_KEY" | ssh-add -
          # Export agent socket for subsequent steps.
          echo "SSH_AUTH_SOCK=$SSH_AUTH_SOCK" >> "$GITHUB_ENV"
          echo "SSH_AGENT_PID=$SSH_AGENT_PID" >> "$GITHUB_ENV"

      - name: Deploy open-seo (pull → migrate → up) (CI-01, CI-02)
        shell: bash
        env:
          VPS_HOST: ${{ secrets.VPS_HOST }}
          VPS_USER: ${{ secrets.VPS_USER }}
        run: |
          set -euo pipefail
          # StrictHostKeyChecking=yes + UserKnownHostsFile= our pinned file (CI-03, NEVER =no).
          SSH_OPTS="-o StrictHostKeyChecking=yes -o UserKnownHostsFile=$HOME/.ssh/known_hosts -o BatchMode=yes -o ConnectTimeout=30"

          ssh $SSH_OPTS "$VPS_USER@$VPS_HOST" bash -s <<'REMOTE'
          set -euo pipefail
          cd "${VPS_REPO_PATH:-/home/deploy/TeveroSEO}"

          echo "[deploy] git pull origin main"
          git fetch origin main
          git checkout main
          git reset --hard origin/main

          echo "[deploy] running migrations BEFORE container swap (CI-02)"
          docker compose -f docker-compose.vps.yml --profile migrate run --rm open-seo-migrate

          echo "[deploy] rebuilding open-seo + open-seo-worker (zero-downtime for other services)"
          docker compose -f docker-compose.vps.yml up -d --build open-seo open-seo-worker

          echo "[deploy] pruning old images"
          docker image prune -f

          echo "[deploy] complete"
          REMOTE

      - name: Verify open-seo health
        shell: bash
        env:
          VPS_HOST: ${{ secrets.VPS_HOST }}
          VPS_USER: ${{ secrets.VPS_USER }}
        run: |
          set -euo pipefail
          SSH_OPTS="-o StrictHostKeyChecking=yes -o UserKnownHostsFile=$HOME/.ssh/known_hosts -o BatchMode=yes -o ConnectTimeout=30"
          # Poll docker healthcheck up to 90s for open-seo + open-seo-worker.
          ssh $SSH_OPTS "$VPS_USER@$VPS_HOST" bash -s <<'REMOTE'
          set -euo pipefail
          cd /home/deploy/TeveroSEO
          for i in $(seq 1 18); do
            OPEN_SEO=$(docker inspect -f '{{.State.Health.Status}}' $(docker compose -f docker-compose.vps.yml ps -q open-seo) 2>/dev/null || echo "missing")
            WORKER=$(docker inspect -f '{{.State.Health.Status}}' $(docker compose -f docker-compose.vps.yml ps -q open-seo-worker) 2>/dev/null || echo "missing")
            echo "[verify] attempt $i: open-seo=$OPEN_SEO worker=$WORKER"
            if [ "$OPEN_SEO" = "healthy" ] && [ "$WORKER" = "healthy" ]; then
              echo "[verify] both services healthy"
              exit 0
            fi
            sleep 5
          done
          echo "[verify] services did not become healthy in 90s"
          docker compose -f docker-compose.vps.yml logs --tail=100 open-seo open-seo-worker
          exit 1
          REMOTE
```

Notes on WHY:
- **`concurrency.cancel-in-progress: false`**: never cancel a deploy mid-flight (could leave containers half-swapped). Serial deploys only.
- **ssh-agent over key-on-disk**: private key is never persisted to the filesystem of the Actions runner.
- **Two SSH invocations (deploy + verify)**: keeps the verify step's exit code isolated from the deploy script so you can see which phase failed.
- **`docker image prune -f`**: prevents the VPS disk from filling with old open-seo image layers over many deploys.
- **Path filter includes `docker-compose.vps.yml`**: if the compose file changes, redeploy is warranted even without an open-seo-main change.
- **`workflow_dispatch`**: allows manual redeploy from the Actions UI (e.g. after fixing the VPS manually).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → VPS SSH | Untrusted network; authenticated by pinned host key (KNOWN_HOSTS) + client private key (ssh-agent) |
| Runner shell → remote bash heredoc | Shell injection surface if any secret is interpolated into the remote script body (we do NOT do this — only `$VPS_USER` and `$VPS_HOST` are used in the local ssh command) |
| `deploy` user → docker daemon | `deploy` is in the `docker` group → root-equivalent on the box; mitigated by scope (only CI can SSH in, key is ed25519-protected) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | S (Spoofing) | MITM on SSH | mitigate | CI-03: `KNOWN_HOSTS` secret populated via `ssh-keyscan -H` + `StrictHostKeyChecking=yes`. Workflow NEVER sets `=no`. Acceptance criteria greps enforce this. |
| T-05-06 | T (Tampering) | Compromised GH Actions runner reads private key | mitigate | Private key loaded via `ssh-agent`, never written to disk. `BatchMode=yes` prevents any interactive prompt-based exfil. Attack surface is the runner OS itself (accepted risk; standard GH-hosted runner). |
| T-05-07 | R (Repudiation) | "who deployed what" | mitigate | GitHub Actions run history immutably records the commit SHA; `git reset --hard origin/main` on VPS means deployed state is always traceable to a commit. |
| T-05-08 | I (Info disclosure) | Secrets in logs | mitigate | All secrets are used via `${{ secrets.* }}` → GitHub masks them in logs. Heredoc body does NOT echo secrets. `set -euo pipefail` fails fast without dumping env. |
| T-05-09 | D (DoS) | Concurrent deploys racing | mitigate | `concurrency.group: deploy-vps-open-seo` with `cancel-in-progress: false` — serial deploys only, never mid-flight cancellation. |
| T-05-10 | E (Elevation) | Attacker with repo write access pushes malicious code | accept | Inherent to any CI/CD. Mitigated out-of-band by branch protection on `main` (configured in GitHub repo settings, not this workflow). Documented in `user_setup`. |
| T-05-11 | I (Info disclosure) | Command injection via `$VPS_HOST`/`$VPS_USER` | mitigate | Secrets are quoted in the ssh command; heredoc body uses `<<'REMOTE'` (single-quoted) → no local variable expansion inside remote script. |
</threat_model>
