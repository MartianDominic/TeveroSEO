<interfaces>
**Postgres image contract** (postgres:16-alpine official image):
- Env vars `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` create the primary DB/user
- Files in `/docker-entrypoint-initdb.d/*.sql` run once on empty data volume (and ONLY once — subsequent starts skip them)
- Therefore init.sql must create the ADDITIONAL databases and users that aren't handled by the standard env vars

**Decision (Claude's discretion, documented here for executor):**
- Primary env-var-managed user = `postgres` superuser (POSTGRES_USER=postgres, POSTGRES_DB=postgres)
- init.sql creates `open_seo` DB owned by role `open_seo_user` and `alwrity` DB owned by role `alwrity_user`
- Passwords are substituted via `\set` from envs `OPEN_SEO_DB_PASSWORD` and `ALWRITY_DB_PASSWORD` passed to the container. Postgres initdb supports psql variable substitution via POSTGRES_INITDB_ARGS is NOT available — use a template approach: init.sql is actually a shell script (`.sh` extension) OR we use plain `.sql` with psql variable substitution from the session env.

Chosen approach: use a `.sh` script — postgres initdb also runs `*.sh` files from that directory. This lets us read `$OPEN_SEO_DB_PASSWORD` and `$ALWRITY_DB_PASSWORD` at initdb time. BUT, per 04-CONTEXT.md the filename is `init.sql`. We honor the CONTEXT.md filename (`init.sql`) and use an alternate pattern: the passwords come from two dedicated env vars that are interpolated by a tiny wrapper `init.sh` which sources them and runs psql. Both files live under `docker/postgres/`.

Actually — the cleanest pattern is to put passwords in envs `OPEN_SEO_DB_PASSWORD`/`ALWRITY_DB_PASSWORD` and have init.sql use psql's `\gexec` + `CURRENT_SETTING` trick is complex. Simpler: Since 04-CONTEXT.md specifies `init.sql`, we use a shell script `init.sh` AS WELL for password injection and keep `init.sql` as the role/DB creation WITHOUT passwords, then grant passwords via `init.sh`. Postgres initdb executes both `*.sql` and `*.sh` files in alphabetical order.

Final executor instruction: Create TWO files — `init.sql` (DB + role creation, no password literals) and `init.sh` (ALTER ROLE ... PASSWORD '$VAR'). 04-04 compose mounts both.

Actually re-reading 04-CONTEXT.md: "docker/postgres/init.sql — creates both open_seo and alwrity DBs if they don't exist" — no mention of passwords being in SQL. So init.sql handles DB+role creation; passwords are injected via the shell companion. This preserves CONTEXT fidelity while being secure.

**Nginx domains:**
- `app.openseo.so` → open-seo-main (port 3001)
- AI-Writer domain: the actual domain will be provided via env var `AI_WRITER_DOMAIN` at deploy time. Default placeholder: `app.alwrity.com` (replace in deploy).

**Redis config contract** (from Phase 3 SUMMARY via 04-CONTEXT.md + BullMQ docs):
- `maxmemory 512mb` — cap at 512MB
- `maxmemory-policy noeviction` — BullMQ REQUIRES this; eviction causes job data loss
- `save 60 1000` — RDB snapshot if ≥1000 keys changed in 60s (mild persistence for BullMQ repeat jobs + audit progress)
- `appendonly no` — RDB only; AOF would double disk I/O and isn't needed for this workload

    ```sql
    -- This file runs ONCE by the postgres:16-alpine image via /docker-entrypoint-initdb.d/.
    -- Passwords are set by the companion init.sh using env vars (keeps secrets out of VCS).

    -- open-seo-main database

    \c alwrity

    ```

    Write `docker/postgres/init.sh` with exactly this content (script to inject passwords at initdb time):

    ```sh
    #!/bin/sh
    set -euo pipefail

    # Require passwords to be provided via env — fail loudly if missing (DOCKER-03).
    : "${OPEN_SEO_DB_PASSWORD:?OPEN_SEO_DB_PASSWORD env var is required for postgres initdb}"
    : "${ALWRITY_DB_PASSWORD:?ALWRITY_DB_PASSWORD env var is required for postgres initdb}"

    psql -v ON_ERROR_STOP=1 --username "$POSTGRES_USER" --dbname "$POSTGRES_DB" <<-EOSQL
      ALTER ROLE open_seo_user WITH PASSWORD '${OPEN_SEO_DB_PASSWORD}';
      ALTER ROLE alwrity_user  WITH PASSWORD '${ALWRITY_DB_PASSWORD}';
    EOSQL

    echo "[init.sh] Passwords set for open_seo_user and alwrity_user"
    ```

    Make init.sh executable: `chmod +x docker/postgres/init.sh`.

    Note: alphabetical init order — `init.sh` runs AFTER `init.sql` because `sh` > `sql` in ASCII. That's what we want (roles must exist before ALTER ROLE).
