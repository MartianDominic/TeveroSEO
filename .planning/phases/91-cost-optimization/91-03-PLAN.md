# Plan 91-03: Batching Optimization

## Objective

Increase batch sizes for LLM operations to reduce API call count by **67-80%**. Grok 4.1 has a 2M token context window — the current 50-keyword limit is artificially conservative.

## Prerequisites

- Plan 91-02 complete (prompt caching active)
- Grok 4.1 Fast API access verified

## Background

| Operation | Current Batch | Optimal Batch | API Call Reduction |
|-----------|---------------|---------------|-------------------|
| Keyword classification | 50 | 200 | 75% |
| ResilientClassifier Pass 2 | 5 | 25 | 80% |
| Translation | 10 | 30 | 67% |
| Funnel classification | 100 | 250 | 60% |

**Token Math for 200-Keyword Batch:**
- System prompt: ~350 tokens (fixed)
- User prompt: ~150 tokens (fixed)
- 200 keywords × 15 tokens: ~3,000 tokens
- **Total input: ~3,500 tokens** (well under 2M context)
- Output per keyword: ~40 tokens
- 200 × 40 = 8,000 output tokens
- **Required maxTokens: 12,000** (with headroom)

## Rollback Plan

If quality issues detected:
1. Revert batch sizes via env vars
2. Gradually increase: 50 → 100 → 150 → 200 with 24h at each step
3. Monitor confidence scores at each step

## A/B Test Protocol

1. Week 1: Run 50% of batches at 50 keywords, 50% at 100
2. Compare: confidence scores, error rates, latency
3. If quality stable: increase to 150
4. Week 2: Run 50% at 150, 50% at 200
5. Settle on highest batch size with stable quality
