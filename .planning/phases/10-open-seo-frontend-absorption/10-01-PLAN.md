<objective>
Create the Next.js server actions layer in `apps/web/src/actions/seo/` that proxy all open-seo API calls to `http://open-seo-api:3001`. These server actions replace TanStack Start `createServerFn()` calls — the ported pages will call these instead of calling open-seo directly from the browser.

Purpose: Establish the server-side API bridge so the browser never makes cross-origin requests to open-seo. All network calls to open-seo happen in Next.js server context via `postOpenSeo` / `getOpenSeo` from the existing `server-fetch.ts`.

Output: Four action files exporting typed async functions. Each function appends `?client_id={clientId}` to its open-seo API call (Phase 10 fallback before Phase 11 Clerk JWT auth).
</objective>

| Next.js server → open-seo-api | Internal Docker network; client_id passed as query param (Phase 10 fallback) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Spoofing | clientId in server action params | mitigate | Validate clientId is non-empty string before forwarding; open-seo enforces client scoping server-side |
| T-10-02 | Information Disclosure | open-seo error messages forwarded to browser | mitigate | Catch errors in server actions; return sanitized message, not raw upstream error body |
| T-10-03 | Elevation of Privilege | client_id bypass on open-seo API | accept | open-seo enforces X-Client-ID scoping (Phase 6/7); query param is belt-and-suspenders for Phase 10 only |
</threat_model>
