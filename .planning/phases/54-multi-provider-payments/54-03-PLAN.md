# Plan 54-03: Webhook Handlers + InvoiceService Updates

## Goal
Create Revolut webhook endpoint and update InvoiceService to handle both providers.

## Tasks

### 1. Revolut Webhook Route (2h)
- [ ] Create `src/routes/api/webhooks/revolut.ts`
  ```typescript
  POST /api/webhooks/revolut
  - Read raw body (not parsed JSON)
  - Verify signature via RevolutProvider.verifyWebhook()
  - Route events to handlers
  - Return 204 on success
  ```
- [ ] Event handlers:
  - `ORDER_COMPLETED` → InvoiceService.handlePaymentSuccess()
  - `ORDER_CANCELLED` → Log activity only
  - `ORDER_PAYMENT_DECLINED` → Log activity, optionally notify
  - Unknown events → Log warning, return 204

### 2. Update InvoiceService (2.5h)
- [ ] Modify `handlePaymentSuccess(externalId, paymentId, provider)`
  - Accept provider parameter ('stripe' | 'revolut')
  - Look up invoice by stripeInvoiceId OR revolutOrderId
  - Update appropriate columns based on provider
  - Same downstream logic (mark paid, update contract, trigger onboarding)

- [ ] Modify `sendToClient(invoiceId, email, name, provider?)`
  - Use PaymentProviderFactory to get correct provider
  - Call provider.createPaymentSession()
  - Store provider-specific IDs in invoice
  - Set `payment_provider` column

- [ ] Add `getInvoiceByRevolutOrderId(orderId)` to InvoiceRepository

### 3. Idempotency Handling (1.5h)
- [ ] Extend idempotency-schema.ts for webhook deduplication
  - `webhook_idempotency` table: `event_id`, `provider`, `processed_at`
- [ ] Check before processing webhook
- [ ] Insert after successful processing
- [ ] Return 204 for duplicates (no error)

### 4. Activity Logging (1h)
- [ ] Add activity types for Revolut events
  - `revolut_order_created`
  - `revolut_payment_completed`
  - `revolut_payment_failed`
- [ ] Log provider in activity data

### 5. Tests (1.5h)
- [ ] Integration tests for webhook route
  - Valid signature → processes event
  - Invalid signature → 403
  - Duplicate event → 204 (idempotent)
- [ ] InvoiceService tests with both providers

## Webhook Security Checklist
- [x] Signature verification before processing
- [x] Timestamp validation (5-minute window)
- [x] Idempotency to prevent duplicate processing
- [x] No sensitive data in logs
- [x] 204 response (no body to leak info)
